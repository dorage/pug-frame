# 작업지시사항

## 모노레포 / 패키지 네이밍

- NX 모노레포로 구성하며, 모든 패키지는 `packages/*`에 둡니다.
- 패키지 이름은 `@pug-frame/*` 스코프를 사용합니다. (예: `@pug-frame/render`, `@pug-frame/cli`, `@pug-frame/canvas`)
- 패키지 디렉터리명은 스코프를 뺀 이름(`render`, `cli`, `canvas`)으로 하고, `project.json`의 `name`도 동일하게 맞춥니다.
- 코어 렌더 로직은 `@pug-frame/render`에 두고, 다른 패키지는 이를 의존해 재사용합니다.

## 문서 작성

- 패키지별 API 문서 및 도메인 정보는 `docs/`에 markdown으로 작성하고, `README.md`에 각 문서 링크를 둡니다.
- **테이블을 사용하지 않습니다.** 테이블은 렌더링 없이 볼 때 가독성과 편집 용이성이 떨어집니다.
- 테이블로 표현하고 싶은 구조는 리스트로 대체하고, 각 항목의 컬럼값은 리스트 항목의 하위 컨텐츠로 반복 작성해 해결합니다.
- **작업 이후에는 `README.md`와 관련 문서(`docs/*`)를 항상 최신 상태로 업데이트합니다.** 문법·API·동작이 바뀌면 해당 문서와 예시도 같은 작업 단위에서 함께 반영합니다.

## Git

- 원자적인 작업단위로 커밋을 남깁니다.
- conventional commit을 사용합니다.
- 브랜치 네이밍도 conventional commit과 동일한 `type/task` 컨벤션을 사용합니다.

## Implementaion Note

작업을 진행하면서 구현 방식이 명세와 달라지거나 해석이 필요한 모든 내용을 기록하는 실시간 노트를 User 루트의 `.impl-note` 디렉터리(`~/.impl-note`) 아래에 브랜치명으로 유지한다.

이미 동일 브랜치 명으로 존재하는 경우, 기존 내용을 유지하면서 작성한다.

### 작성 규칙

다음 내용이 포함되어야 한다.

- 설계 결정: 지시가 모호하여 본인이 내린 선택들
- 편차: 의도적으로 명세를 따르지 않은 부분과 그 이유
- 트레이드오프: 고려했던 대안들과 현재 방식을 선택한 이유
- 미결 질문: 확인이나 수정이 필요한 사안들
- 누락된 도메인 지식: 현재 리포지토리에서 찾을 수 없었던 도메인 지식
- 사용자의 선택: 주어졌던 선택지와 선택된 내용

### 템플릿

```` markdown
# Implementation Note: {작업명} ({단계})

---

## 작성방식

### 1. 설계 결정

Draft 가 모호하여 내린 선택들. 각 항목은 `D{n}. {요약}` 형태로 일련번호를 부여한다. 한 번 부여한 번호는 폐기되더라도 결번으로 남겨 외부 참조가 깨지지 않게 한다.

#### D1. <요약 제목>

- **결정**: <무엇을 어떻게 정했는지 한 줄>
- **이유**:
  - <사실/제약/원칙 기반>
- **상위 대안**: <더 무거운 대안과 채택 트리거 조건. 단순한 결정이면 생략>

### 2. 편차 (Draft → Thinking)

의도적으로 Draft 명세를 따르지 않은 항목. `D-{n}` 형태로 일련번호를 부여하고, 상세 근거는 위 설계 결정의 해당 `D{n}` 항목을 참조한다.

- **D-1**: <한 줄 요약> — 근거: D1

### 3. 트레이드오프

`{선택안} vs {대안}` 형식. 비용/이득은 `↑`/`↓` 로 한눈에 비교 가능하게.

- **<선택안> vs <대안>**: 선택안 → <장점> ↑, <단점> ↓. 대안 → <장점> ↑, <단점> ↓.

### 4. 미결 질문

확인이나 수정이 필요한 사안. `Q{n}` 형태로 일련번호 부여. 해소되면 항목 자체를 지우지 말고 `**해소**:` 불릿을 추가한다.

- **Q1**: <질문> — <누구/언제 결정>
  - **해소**: <결정 내용>

### 5. 누락된 도메인 지식 *(있을 때만)*

Wiki 에서 찾을 수 없어 별도로 가정/조사한 도메인 지식. 작업 후 Wiki 에 반영해야 할 항목으로 추적한다.

- <항목> — Wiki 반영 대상: `specs/wiki/<file>.md`

### 6. 사용자의 선택 *(있을 때만)*

여러 선택지를 제시하고 사용자가 결정한 항목. 제시한 선택지 전체와 채택안, 사유를 함께 적는다.

- **<선택 주제>**
  - 제시안: A / B / C
  - 채택: B
  - 사유: <한 줄>

### 1. 추가 결정 (Plan 대비 편차 또는 보강)

`DI{n}` 형태로 일련번호. 형식은 "설계 결정" 과 동일.

#### DI1. <요약 제목>

- **결정**:
- **이유**:

### 2. 검증 결과

구현 후 실제로 통과한 검증을 사실 위주로 나열. 명령어·결과, 수동 검증 시의 응답·상태 코드 등.

- `pnpm build` — 통과
- `pnpm test:regression` — 통과 (<N> tests)
- `pnpm test:unit` — 통과 (<N> tests)
- 수동: <시나리오> — <결과>

### 3. 알려진 비차단 경고 *(있을 때만)*

동작에 영향 없으나 향후 처리할 경고/이슈. 무엇이 떴는지, 왜 지금은 무시해도 되는지를 함께 적는다.

- <경고> — 이유: <왜 비차단인지>

### 4. 트레이드오프 (구현 중 재확인) *(있을 때만)*

Thinking 단계와 별개로, 구현 중 의식적으로 비용을 받아들인 결정.
